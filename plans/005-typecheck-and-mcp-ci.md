## Why this matters

Two cheap, high-value gaps in the developer/CI feedback loop:

1. The frontend has `strict: true` TypeScript but **no `typecheck` script** — the
   only way to gate types locally is a full `next build`, which is slow and writes
   `.next/`. A `tsc --noEmit` script gives a fast local type gate (once the project
   has been built/run once — see the ordering note).
2. CI (`.github/workflows/build.yml`) builds, lints, and E2E-tests **only**
   `dns-manager/`. The MCP server (`mcp-server/`, TypeScript) is **never built or
   typechecked in CI**, so type errors and broken imports there can land on `main`
   undetected. The MCP server is the AI-agent integration surface — it should at
   least compile in CI.

This plan adds the `typecheck` script and a CI job that builds + typechecks the
MCP server. Low risk (additive), and it makes plans 003/004's TypeScript changes
enforceable.

## Why the ordering matters (read before editing CI)

`dns-manager/tsconfig.json` `include` references `next-env.d.ts` and
`.next/types/**/*.ts`. Both are **generated by `next build` / `next dev`** and are
**gitignored** (`dns-manager/.gitignore` lists `next-env.d.ts`). In a fresh
checkout (or a fresh CI runner) they do not exist, so `tsc --noEmit` cannot resolve
Next.js ambient types and floods with errors like `Cannot find module 'next'`,
`Cannot find module 'next/font/google'`, `Could not find a declaration file for
module 'lucide-react'`. **These are not real type errors** — they vanish once
`next build` has generated `next-env.d.ts`. Verified during execution: the main
repo, after a build, passes `tsc --noEmit` with **0 errors**.

Consequence: the CI `Typecheck` step **must run after** the existing `TypeScript
build` step (which invokes `next build` and generates `next-env.d.ts`). The
existing build step already type-checks via Next; this explicit `tsc --noEmit`
step is belt-and-suspenders (it also covers files outside the Next build graph,
e.g. test files), and the bigger payoff is the local `npm run typecheck` script.

## Steps

### Step 1: Add the `typecheck` script

In `dns-manager/package.json` `scripts`, add:
```json
    "typecheck": "tsc --noEmit"
```
**Verify**: `cd dns-manager && npm ci && npm run build && npm run typecheck` exits
0 (note: `npm run build` first — it generates `next-env.d.ts`, without which
typecheck reports spurious "Cannot find module 'next'" errors; see the ordering
note). If `npm run typecheck` reports genuine type errors **after** a successful
`npm run build`, STOP and report them — do not fix unrelated code in this plan.

### Step 2: Add a frontend typecheck step in CI — AFTER the build

In `.github/workflows/build.yml`, add this step **immediately after** the
`TypeScript build` step and **before** `Lint` (so `next-env.d.ts` already exists):
```yaml
      - name: Typecheck
        working-directory: dns-manager
        # Runs after "TypeScript build" because tsc needs next-env.d.ts, which
        # `next build` generates (it is gitignored and absent on a fresh checkout).
        run: npm run typecheck
```

> Do NOT place this step before `TypeScript build` — it will fail on a fresh
> runner because `next-env.d.ts` does not exist yet.

### Step 3: Add an MCP server build job

Append a second job to `.github/workflows/build.yml` (sibling of `build:`), so the
MCP server compiles in CI:
```yaml
  mcp-server:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: mcp-server/package-lock.json

      - name: Install dependencies
        working-directory: mcp-server
        run: npm ci

      - name: Build (tsc — fails on type errors)
        working-directory: mcp-server
        run: npm run build
```

`mcp-server/package-lock.json` exists (confirmed), so `npm ci` is correct here.

**Verify**: locally reproduce the job — `cd mcp-server && npm ci && npm run build`
exits 0.

### Step 4: Confirm the whole frontend pipeline still passes locally

**Verify**: `cd dns-manager && npm run build && npm run typecheck && npm run lint`
all exit 0 (build first, for the reason above).
