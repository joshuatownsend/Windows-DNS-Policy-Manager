# E2E Tests with Playwright — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Playwright E2E tests covering all 9 tabs with a mock bridge, integrated into the existing CI workflow.

**Architecture:** Use Playwright's `route()` API to intercept all `/api/*` requests and return JSON fixtures. No real bridge or DNS server needed. Tests exercise the full React UI → Zustand store → API client → mock response cycle. Each tab gets its own test file.

**Tech Stack:** Playwright, Next.js dev server, JSON fixtures.

---

## Design: Mock Bridge Strategy

Instead of running the PowerShell bridge, use Playwright's network interception:

```ts
// In each test, mock the bridge responses
await page.route('/api/health', (route) =>
  route.fulfill({ json: { success: true, status: 'ok', dnsModuleAvailable: true } })
);
```

Fixture data lives in `e2e/fixtures/` as JSON files. A shared helper `mockBridge(page)` sets up common routes (health, zones, servers).

### What we test:
- Navigation between all 9 tabs
- Server CRUD (add, edit, test, remove)
- Zone browsing and record filtering
- Policy list, toggle, delete
- Create Policy form + command generation
- Wizard scenario completion (at least geo-location end-to-end)
- Backup export flow
- PowerShell output display
- Help panel open/close/navigation
- DNSSEC zone signing status display
- Bridge offline → graceful degradation

### What we don't test:
- Actual DNS operations (that's integration testing against a real server)
- Bridge PowerShell logic (tested separately if needed)
